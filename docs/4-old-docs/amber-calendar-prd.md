> 版本：v1.1  
> 更新时间：2026-05-08  
> 状态：Phase 1 桌面 Web MVP 初版已完成，进入稳定化与验收阶段

---

## 1. 产品概述

**产品名称**：Amber Calendar

**产品定位**：一款面向桌面 Web 的现代中国日历与传统历法工具。以公历农历对照、老黄历查询为核心，未来扩展至八字排盘、择日、每日洞察等命理参考功能。产品风格追求现代、克制、温暖、专业，强调文化底蕴但拒绝陈旧的算命网站美学。

**目标用户需求**：
- 快速查询某一天的公历、农历、节气、节日信息
- 查看每日黄历宜忌、冲煞、吉凶神煞等传统历法参考
- 未来：根据出生日期推算八字命理、筛选吉日、获取每日行动建议

**核心产品价值**：
- 将传统历法信息以现代、可信、易读的方式呈现
- 所有数据基于 lunar-javascript 提供的历法与黄历规则在客户端本地生成，无外部 API 依赖
- 模块化架构，功能之间相互独立，不混杂在同一界面中

## 3. 目标用户

### 主要用户画像

- 对中国传统文化有兴趣的华语互联网用户
- 需要查询农历日期、节气、黄历宜忌的日常使用者
- 关注婚嫁、搬家、开业等重要日期选择的用户
- 对八字命理感兴趣但希望使用现代、干净工具的用户

### 使用场景

- 日常查看今天的农历日期和黄历信息
- 为重要事项（婚嫁、搬家、开业）查询某天的宜忌
- 浏览某个月的节气和传统节日
- （未来）输入出生日期时间，查看八字排盘
- （未来）根据需求筛选适合的吉日

### 用户动机

- 实用性：快速获取传统历法信息
- 参考性：为决策提供文化层面的参考
- 文化认同：使用一个尊重传统但不迷信的现代工具

## 6. 模块分解

### 6.1 日历黄历（Calendar + Huangli）

- **用途**：公农历对照 + 每日黄历查询
- **当前状态**：✅ MVP 初版已完成，进入稳定化阶段
- **MVP 需求**：月历浏览、日期选择、黄历详情展示、7 种交互状态、桌面 Web 布局
- **未来需求**：年份快速跳转、日期搜索、法定节假日标注、周视图、移动端浏览器适配
- **不在范围内**：记事/日程功能、日历订阅/导出

### 6.2 四柱八字（Four Pillars / BaZi）

- **用途**：根据出生年月日时推算天干地支四柱，分析五行与十神
- **当前状态**：🔲 占位页面，逻辑未实现
- **MVP 需求**：
  - 输入：出生日期、出生时辰（时柱）、性别（影响大运顺逆）
  - 输出：年柱、月柱、日柱、时柱的天干地支
  - 展示：五行属性、十神关系、日主强弱初步判断
- **必须考虑的技术问题**：
  - 月柱以节气（节）为分界，不以农历月初一为准
  - 早子时（00:00–01:00）与晚子时（23:00–00:00）的归属问题
  - 真太阳时（True Solar Time）选项：不同经度的出生地可能影响时柱判定
  - 出生地与时区：用户输入的出生时间应明确是北京时间还是当地时间
  - 大运起排：根据性别和年干阴阳判定顺逆，计算大运起始年龄
  - `lunar-javascript` 提供 `EightChar` 类，需验证其对以上边界情况的处理
- **未来需求**：大运流年排盘、五行强弱图表、藏干分析、神煞标注
- **不在范围内**：AI 解读、付费详批、流年运势预测文本生成

### 6.3 择日（Date Selection）

- **用途**：根据事项类型和黄历宜忌筛选推荐吉日
- **当前状态**：🔲 占位页面，逻辑未实现
- **MVP 需求**：选择事项类型（婚嫁/搬家/开业等）→ 设定日期范围 → 根据黄历宜忌自动筛选 → 输出推荐日期列表
- **未来需求**：结合个人八字冲煞过滤（依赖八字模块）、推荐结果详情查看
- **不在范围内**：社交分享、专家推荐、付费择日服务

### 6.4 每日洞察（Daily Insight）

- **用途**：将传统历法数据转化为简洁易懂的每日行动提示
- **当前状态**：🔲 占位页面，逻辑未实现
- **MVP 需求**：基于当日建除十二神、黄道/黑道、宜忌条目数量生成综合每日摘要；展示当日节气物候提示；高亮重要宜忌事项
- **未来需求**：个性化推送（需后端支持）、基于八字的个人化建议
- **不在范围内**：AI 生成内容、星座运势、社交功能

### 6.5 设置（Settings）

- **用途**：管理应用偏好和显示配置
- **当前状态**：🔲 占位页面，逻辑未实现
- **MVP 需求**：深色/浅色模式切换、数据来源与算法说明页、关于本产品页
- **未来需求**：时区设置（解除 Asia/Shanghai 硬编码）、语言选择（简体/繁体）、日历起始日设置
- **不在范围内**：用户账户管理、云端同步、数据导出

## 7. 数据与算法需求

### 7.1 数据来源

所有历法和黄历数据基于 `lunar-javascript` 库提供的历法规则与黄历规则，在客户端本地实时生成。不依赖任何外部 API 或预计算数据库。`lunar-javascript` 的算法涵盖农历推算、节气计算、干支历法、宜忌规则等，但不同黄历流派可能存在差异。

### 7.2 适配器模式

UI 组件**不得直接调用** `lunar-javascript`。所有数据访问必须通过适配器层完成：

```
UI Components → CalendarContext → IAlmanacProvider → LunarJsAdapter → lunar-javascript
```

- `IAlmanacProvider`：定义数据接口契约
- `LunarJsAdapter`：实现具体的 lunar-javascript 调用和数据映射
- `types.ts`：定义 `CalendarDay`、`AlmanacDay` 等数据类型

这一架构保证：如果未来更换数据源，只需替换 Adapter，不影响 UI 层。

### 7.3 时区假设

当前所有日期计算基于 `Asia/Shanghai` (UTC+8) 时区。UI 中不显示时区选择器。这一假设在未来的设置模块中可以解除。

### 7.4 数据限制与免责

| 规则 | 说明 |
|:---|:---|
| 无硬编码数据 | 所有黄历字段必须来自实时计算，禁止在组件中写入固定的宜忌文本 |
| 免责声明 | 每个展示黄历数据的面板底部必须包含数据来源和"仅供参考"声明 |
| 不同传统差异 | 不同黄历流派（如通胜、协纪辨方书）对宜忌的判定可能不同。当前使用 lunar-javascript 的默认算法，不声称覆盖所有流派 |
| 措辞规范 | 禁止使用"准确""权威""科学验证"等绝对性措辞。统一使用"参考""传统算法""本地生成" |

### 7.5 数据验证状态

| 字段 | lunar-javascript 来源方法 | 验证状态 | 缺失时 UI 行为 |
|:---|:---|:---|:---|
| 公历日期 | `Solar.fromYmd()` → `.getYear()` `.getMonth()` `.getDay()` | ✅ 已验证 | 不会缺失 |
| 农历日期 | `solar.getLunar()` → `.getMonthInChinese()` `.getDayInChinese()` | ✅ 已验证 | 不会缺失 |
| 节气 | `lunar.getJieQi()` | ✅ 已验证 | 无节气时不显示，回退显示农历日 |
| 节日 | `solar.getFestivals()` + `lunar.getFestivals()` | ⚠️ 需更多测试 | 无节日时不显示，回退显示农历日 |
| 宜 | `lunar.getDayYi()` | ✅ 已验证 | 返回空数组时显示"无" |
| 忌 | `lunar.getDayJi()` | ✅ 已验证 | 返回空数组时显示"无" |
| 冲 | `lunar.getDayChongDesc()` | ✅ 已验证 | 返回空字符串时显示"—" |
| 煞 | `lunar.getDaySha()` | ✅ 已验证 | 返回空字符串时显示"—" |
| 建除十二神 | `lunar.getZhiXing()` | ✅ 已验证 | 不会缺失 |
| 值神 | `lunar.getDayTianShen()` + `.getDayTianShenType()` | ⚠️ 需更多测试 | 返回空时显示"—" |
| 五行纳音 | `lunar.getDayNaYin()` | ⚠️ 需更多测试 | 返回空时显示"—" |
| 吉神宜趋 | `lunar.getDayJiShen()` | ⚠️ 需更多测试 | 返回空数组时显示"无" |
| 凶神宜忌 | `lunar.getDayXiongSha()` | ⚠️ 需更多测试 | 返回空数组时显示"无" |

**说明**：
- "已验证"指通过多个样本日期（2024-02-04、2024-02-10、2024-12-31、2025-01-01、当天）确认返回值格式和内容合理
- "需更多测试"指基本格式已确认，但尚未覆盖边界日期（如闰月、岁末、节气当天）的完整验证
- 所有字段当前均未发现返回 `null` 或 `undefined` 的情况，但防御性处理仍应保留

## 8. UX 与设计原则

### 8.1 视觉方向

Amber Calendar 的视觉语言是**现代东方历书**与**每日洞察工具**的融合：

- **温暖但不陈旧**：暖纸色背景 (`#eae6dd`)、墨色文字 (`#1a1815`)，营造书卷气
- **文化但不装饰**：通过衬线字体（Noto Serif SC）和配色传递中国文化基因，不使用纹理、插图、边框装饰
- **克制的强调色**：朱砂 (`#c4553e`) 仅用于选中状态、忌讳标记和关键强调。黛绿 (`#4e7a5e`) 用于宜/吉方向。铜赭 (`#a07850`) 用于周末等次要标记
- **信息密度优先**：宁可紧凑也不留白，但保持呼吸感

### 8.2 布局原则

- 桌面端：统一白色面板内左右分栏（日历 60% / 黄历 40%），右侧 76px 模块导航
- 面板之间用 `border` 分隔，不用阴影浮动
- 整体感觉是"一本打开的书"，不是"多个浮动卡片"

### 8.3 字体与密度

| 层级 | 字体 | 字号范围 | 用途 |
|:---|:---|:---|:---|
| 一级 | Noto Serif SC, 300 weight | 2.75rem | 黄历面板日期数字 |
| 二级 | Noto Serif SC, 500 weight | 1.125rem | 月份标题、模块标题 |
| 三级 | Inter / 系统黑体 | 0.875rem | 正文、Tag 文本、元数据值 |
| 四级 | Inter | 0.8125rem | 副文本、神煞、宜忌 chip |
| 五级 | Inter | 0.6875rem–0.75rem | 标签、分类名、脚注 |

行距统一使用 `1.5`–`1.6`，中文内容适当放宽。

### 8.4 必须避免

- ❌ 传统算命网站的视觉风格（大红大金、密集花边、动态闪烁）
- ❌ 通用后台管理仪表盘风格（蓝色主题、卡片阴影、数据图表）
- ❌ 节日海报风格（大面积插图、节日主题色覆盖）
- ❌ 过度留白的"概念设计"感（内容太少、页面太空）
- ❌ 大量使用亮红色、亮绿色、纯黑色

## 9. 技术需求

### 9.1 技术栈

| 层级 | 选型 | 说明 |
|:---|:---|:---|
| 框架 | React 18+ | 组件化 UI |
| 构建 | Vite | 快速开发和构建 |
| 语言 | TypeScript (strict mode) | 类型安全 |
| 样式 | CSS Modules | 组件级作用域样式，无全局污染 |
| 状态管理 | React Context | 轻量级，当前足够 |
| 历法计算 | lunar-javascript | 天文算法本地计算 |
| 图标 | lucide-react | 轻量级 SVG 图标库 |
| 工具 | classnames | 条件 CSS class 组合 |

### 9.2 架构原则

- **模块分离**：每个产品模块（日历、八字、择日等）是独立的组件子树，不共享内部状态
- **适配器模式**：数据层通过 `IAlmanacProvider` 接口隔离，UI 不直接依赖 `lunar-javascript`
- **类型安全**：所有数据结构通过 `types.ts` 定义，启用 `verbatimModuleSyntax`
- **路径别名**：使用 `@/*` 映射 `src/*`，保持 import 路径简洁
- **无后端**：当前阶段所有逻辑在客户端完成
- **无路由库**：模块切换通过 `useState` 管理，后续可接入 React Router

## 12. 风险与开放问题

### 12.1 算法准确性

`lunar-javascript` 是目前最成熟的开源中国历法 JavaScript 库之一，但不同黄历流派（如《通胜》《协纪辨方书》《钦定万年历》）对宜忌的判定可能存在差异。产品应明确标注数据来源和算法局限，不声称覆盖所有传统流派。

### 12.2 时区与真太阳时

- 当前固定 `Asia/Shanghai`，不支持其他时区
- 八字排盘在传统上使用"真太阳时"（True Solar Time），而非标准时区时间。这在 Phase 2 开发时需要专门研究和处理
- 不同地区（如新疆、东南亚）的真太阳时与北京时间差异较大，可能影响八字结果

### 12.3 黄历传统差异

不同地区和文化圈对黄历的理解和使用方式不同：
- 中国大陆、台湾、香港、东南亚华人社区可能有不同偏好
- 部分用户可能期望看到特定流派的算法结果
- 产品应保持中性态度，不偏向任何特定流派

### 12.4 信任边界

- 黄历和命理信息具有文化和民俗属性，不是科学预测
- 产品必须通过免责声明、措辞规范、视觉设计传递"参考工具"而非"权威预测"的定位
- 避免用户因过度依赖产品信息做出不当决策

### 12.5 商业化

- 当前阶段不考虑商业化
- 未来可能的方向：高级功能付费（如详细八字解读）、去广告会员、API 服务
- 商业化模式待产品价值验证后再决定

### 12.6 性能

- `lunar-javascript` 库体积约 500KB (gzipped ~167KB)，是当前最大的依赖
- 后续应通过 dynamic import 做 code splitting
- 月历 42 天的批量计算目前是同步的，如果未来增加更复杂的计算（八字大运等），需要考虑 Web Worker 或防抖处理
