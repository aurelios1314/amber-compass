# Amber Compass 产品与界面规则

本文档用于约束后续开发，避免主流程页面反复加入说明、徽章、声明、冗余标签和不必要的视觉噪音。

## 1. 主流程优先服务“使用”

主页面的第一目标是让用户直接完成操作，而不是阅读说明。

用户进入页面后，应能快速完成：

- 查看日历黄历
- 输入生辰信息
- 查看四柱八字与大运
- 切换功能模块

不要把主页面做成说明书、报告页或技术文档页。

## 2. 说明内容放到设置或文档

以下内容不应出现在日历黄历、四柱八字等主工作页面中：

- 长篇免责声明
- 算法原理
- 数据来源
- 外部校验过程
- 技术限制
- 原型说明
- TODO / 技术积压
- 大段帮助文案

如确有必要，应放入：

- 设置
- 数据来源与算法说明
- 帮助 / 关于
- docs 文档
- 专门说明页

主页面只允许极短提示，例如：

- 仅供文化参考
- 结果用于排盘参考
- 详细说明见设置

## 3. 不添加无意义标签和提示符

主流程中不要反复添加装饰性标签，例如：

- 原型
- 即将
- MVP
- CONTROL
- PROTOTYPE
- 测试中
- 验证中

除非能直接帮助用户操作，否则不要放在主界面。

不要出现单独一行的 `?`、`i` 或提示符号。  
如果确实需要 tooltip，必须与标题同行，尺寸小、样式克制、不换行、不破坏排版。

## 4. 所有模块使用统一工作区结构

Amber Compass 的所有模块必须保持统一结构：

- 顶部品牌栏
- 左侧主内容区
- 右侧辅助 / 输入 / 详情区
- 最右侧模块切换栏

日历黄历、四柱八字、择日、每日洞察等模块，只能替换左右区域内的内容，不允许创建独立页面骨架。

不允许：

- 单个模块自建布局
- 改变左右分割线
- 改变导航栏位置
- 破坏 AppShell 结构
- 引入浏览器级全局滚动

## 5. 优先用结构解决问题，而不是加解释

未来 UI 优化目标是：

- 更清楚
- 更紧凑
- 更好扫视
- 更少滚动
- 更少干扰
- 更少说明文字

能通过结构、层级、对齐、命名解决的问题，不要用长文案解决。

## 6. 四柱八字页面规则

四柱八字主页面优先展示：

- 四柱
- 日主
- 十神
- 藏干
- 纳音
- 大运

但不展示：

- 大段算法说明
- 外部验证细节
- 真太阳时技术讨论
- 均时差 TODO
- 复杂免责声明
- 命运断语

专业说明统一进入设置或文档。

干支流通可视化属于高级层，不应默认显示在主排盘页面；如需展示，应通过设置或专业视图开启。

## 7. 大运与滚动规则

大运前 8 步属于核心结果，应尽量在桌面端一屏内完整展示。

原则：

- 优先 4 × 2 网格
- 不使用横向滚动
- 不隐藏核心数据
- 当前大运可轻微高亮，但不要过度装饰

页面滚动处理优先级：

1. 压缩冗余留白
2. 优化组件密度
3. 使用局部滚动
4. 最后才允许页面滚动

不允许因为排版松散，导致用户必须上下左右滚动才能查看基础结果。

## 8. 后续开发禁忌

除非明确要求，否则不要：

- 改 AppShell
- 改模块导航栏结构
- 改左右面板比例
- 把八字输入区移回左侧
- 恢复长说明区域
- 恢复原型 / 即将等标签
- 给大运加横向滚动
- 在主界面增加技术声明
- 将设置页内容塞回主流程

## 9. 总原则

Amber Compass 应该像一个安静、专业、克制的桌面工具。

主页面负责使用。  
设置页负责说明。  
文档负责记录。  
不要把三者混在一起。
